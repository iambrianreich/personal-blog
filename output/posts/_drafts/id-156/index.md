Smarter folks than I have written dozens of chapters and recorded hours of videos about why Pull Requests suck.

They argue that real-time, collaborative programming activities like pair programming and mob programming solve the same problems more efficiently. Or that, simply, you should strive to work in an organization where engineers are so high caliber, and the CI/CD pipeline so pristine, as to render peer review unnecessary.

And I can't not argue against it. Let's assume for the sake of this article that your team does Pull Requests. They find value in having an asynchronous process in which other developers verify an author's code.

And now, you have to navigate the technical and interpersonal landmines that litter the path of Pull Requests.
